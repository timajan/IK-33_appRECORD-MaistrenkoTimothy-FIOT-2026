# Лабораторна робота №4

**З дисципліни:** «WEB-орієнтовані технології. Backend розробки»  
**Тема:** «Розширені можливості Node.js-додатків: логування, завантаження файлів, моніторинг продуктивності»

## Мета роботи

Метою лабораторної роботи є вивчення розширених можливостей серверних застосунків на базі Node.js та Express. У межах роботи необхідно набути практичних навичок реалізації логування HTTP-запитів і подій, організації завантаження файлів на сервер, обробки помилок, вимірювання часу відповіді сервера та моніторингу продуктивності застосунку.

Також метою роботи є вдосконалення вже створеного REST API для системи оренди велосипедів шляхом додавання механізмів логування, завантаження аватара користувача, збереження логів у файли та створення endpoint для перегляду стану сервера.

## Короткі теоретичні відомості

Node.js є платформою для створення серверних застосунків на JavaScript. У поєднанні з фреймворком Express він дозволяє швидко створювати REST API, обробляти HTTP-запити, підключати middleware та реалізовувати серверну логіку.

Одним із важливих елементів серверного застосунку є логування. Логування — це процес запису інформації про події, які відбуваються в системі. До таких подій належать HTTP-запити, помилки, дії користувачів, час виконання запитів та інші службові повідомлення. Логування допомагає аналізувати роботу сервера, знаходити помилки та відстежувати проблеми продуктивності.

Для логування HTTP-запитів у Node.js часто використовується бібліотека **Morgan**. Вона автоматично записує інформацію про метод запиту, URL, статус відповіді та час обробки. Для більш гнучкого логування використовується **Winston**, який дозволяє створювати логи різних рівнів, наприклад `info` та `error`, а також зберігати їх у файли.

Ще одним важливим завданням серверного застосунку є завантаження файлів. У Express для цього використовується middleware **Multer**. Він дозволяє обробляти запити у форматі `multipart/form-data`, приймати файли від користувача, перевіряти їх тип і розмір, а також зберігати на сервері.

Моніторинг продуктивності дає змогу оцінити стан застосунку під час роботи. До основних показників належать `uptime`, використання оперативної пам’яті, CPU та час відповіді сервера. У Node.js ці показники можна отримати за допомогою вбудованого об’єкта `process`.

## Постановка задачі

У межах лабораторної роботи необхідно розширити функціональність уже створеного REST API для системи оренди велосипедів. Основна увага приділяється не створенню нових бізнес-сутностей, а покращенню технічної частини серверного застосунку.

Необхідно реалізувати:

- логування HTTP-запитів за допомогою Morgan;
- файлове логування подій за допомогою Winston;
- логування помилок у файл;
- middleware для обробки помилок;
- middleware для вимірювання часу відповіді сервера;
- endpoint `/status` для перегляду стану сервера;
- завантаження аватара користувача;
- валідацію типу та розміру зображення;
- збереження аватарів у папку `uploads/avatars`;
- віддачу завантажених файлів як статичних ресурсів;
- можливість запуску застосунку через PM2.

Практична реалізація виконується в межах існуючого застосунку `rentBikeApi`, який уже містить маршрути автентифікації, модель користувача, middleware авторизації та підключення до бази даних.

## Опис структури даних

Для реалізації завантаження аватара було використано таблицю `users`, яка вже застосовувалась у попередній лабораторній роботі для реєстрації та авторизації користувачів.

До таблиці `users` було додано нове поле `avatar`. Це поле зберігає шлях до файлу аватара користувача на сервері. Сам файл не зберігається безпосередньо в базі даних, оскільки це не є оптимальним підходом. У базі зберігається лише текстовий шлях, наприклад:

```text
uploads/avatars/user-1-1715000000000.jpg
```

Такий підхід дозволяє зменшити навантаження на базу даних і спростити доступ до файлів через сервер.

Структура таблиці `users` після оновлення містить такі основні поля:

- `id` — унікальний ідентифікатор користувача;
- `name` — ім’я користувача;
- `email` — email користувача;
- `password` — хешований пароль;
- `role` — роль користувача;
- `avatar` — шлях до файлу аватара;
- `created_at` — дата створення користувача.

Для оновлення вже існуючої таблиці було використано SQL-запит:

```sql
ALTER TABLE users ADD COLUMN avatar VARCHAR(255) NULL;
```

## Реалізація логування HTTP-запитів

Для логування HTTP-запитів у застосунок було підключено бібліотеку Morgan. Вона використовується як middleware у головному файлі сервера. Morgan автоматично фіксує всі запити до API, зокрема метод, URL, код відповіді та час виконання.

У файлі `server.js` було додано підключення Morgan:

```js
const morgan = require('morgan');

app.use(morgan('combined'));
```

Формат `combined` дозволяє отримати детальну інформацію про HTTP-запити. Це корисно під час тестування API, оскільки в консолі одразу видно, який endpoint був викликаний і який статус повернув сервер.

## Реалізація файлового логування через Winston

Для збереження логів у файли було використано бібліотеку Winston. На відміну від Morgan, який переважно використовується для HTTP-запитів, Winston дозволяє логувати події різних рівнів і зберігати їх у різні файли.

У проєкті було створено окремий файл `utils/logger.js`, у якому налаштовано logger. Для зручності було використано два основні файли логів:

- `app.log` — для інформаційних повідомлень;
- `error.log` — для помилок.

Приклад налаштування logger:

```js
const winston = require('winston');

const logger = winston.createLogger({
  level: 'info',
  format: winston.format.combine(
    winston.format.timestamp(),
    winston.format.json()
  ),
  transports: [
    new winston.transports.File({ filename: 'logs/app.log', level: 'info' }),
    new winston.transports.File({ filename: 'logs/error.log', level: 'error' })
  ]
});

module.exports = logger;
```

Такий підхід дозволяє зберігати події у структурованому форматі JSON. Завдяки цьому логи легше аналізувати, фільтрувати та використовувати для пошуку помилок.

## Middleware для вимірювання часу відповіді

Для аналізу продуктивності застосунку було створено middleware, який вимірює час обробки кожного HTTP-запиту. На початку запиту зберігається поточний час, а після завершення відповіді обчислюється різниця між початковим і кінцевим часом.

Приклад реалізації:

```js
const logger = require('../utils/logger');

const performanceMiddleware = (req, res, next) => {
  const start = Date.now();

  res.on('finish', () => {
    const duration = Date.now() - start;

    logger.info({
      method: req.method,
      url: req.originalUrl,
      status: res.statusCode,
      responseTime: `${duration}ms`
    });
  });

  next();
};

module.exports = performanceMiddleware;
```

Цей middleware дозволяє бачити, які запити виконуються швидко, а які потребують більше часу. У реальних проєктах така інформація допомагає знаходити повільні endpoints і оптимізувати роботу сервера.

## Обробка помилок

Для централізованої обробки помилок було створено окремий middleware. Його завдання полягає в тому, щоб перехоплювати помилки, записувати їх через Winston і повертати клієнту зрозумілу JSON-відповідь.

Приклад middleware:

```js
const logger = require('../utils/logger');

const errorMiddleware = (err, req, res, next) => {
  logger.error({
    message: err.message,
    method: req.method,
    url: req.originalUrl
  });

  res.status(err.status || 500).json({
    error: err.message || 'Internal Server Error'
  });
};

module.exports = errorMiddleware;
```

Для перевірки роботи обробника помилок було створено тестовий маршрут `/test-error`, який штучно генерує помилку. Завдяки цьому можна переконатися, що помилка коректно записується у файл `error.log`, а користувач отримує JSON-відповідь.

## Реалізація завантаження аватара користувача

У лабораторній роботі завантаження файлів було реалізовано не як окреме абстрактне завантаження, а як практична функція для користувача — оновлення аватара профілю.

Для цього в маршрутах авторизації було створено endpoint:

```text
PATCH /auth/profile/avatar
```

Цей маршрут є захищеним, тому доступ до нього має лише авторизований користувач. Для перевірки користувача використовується `authMiddleware`, який отримує токен із заголовка `Authorization` і додає інформацію про користувача в `req.user`.

Для завантаження файлу використовується Multer. Файл передається у полі `avatar`, після чого зберігається в папку:

```text
uploads/avatars/
```

Для кожного файлу формується унікальна назва, яка містить id користувача та поточний час. Це дозволяє уникнути конфліктів між назвами файлів.

Приклад назви файлу:

```text
user-1-1715000000000.jpg
```

Також було реалізовано валідацію файлів. Дозволено завантажувати лише зображення форматів:

- JPG;
- JPEG;
- PNG;
- WEBP.

Максимальний розмір файлу обмежено до 2 MB.

## Реалізація endpoint `/status`

Для моніторингу стану сервера було створено endpoint:

```text
GET /status
```

Він повертає базову інформацію про роботу застосунку:

- час безперервної роботи сервера;
- використання оперативної пам’яті;
- використання CPU.

Для отримання цих даних використовується вбудований об’єкт `process`.

Приклад реалізації:

```js
router.get('/status', (req, res) => {
  const memoryUsage = process.memoryUsage();
  const uptime = process.uptime();
  const cpuUsage = process.cpuUsage();

  res.json({
    uptime,
    memoryUsage,
    cpuUsage
  });
});
```

Приклад відповіді сервера:

```json
{
  "uptime": 120.45,
  "memoryUsage": {
    "rss": 52342784,
    "heapTotal": 9998336,
    "heapUsed": 7348872,
    "external": 1523360
  },
  "cpuUsage": {
    "user": 125000,
    "system": 46000
  }
}
```

Цей endpoint дозволяє швидко оцінити стан сервера без використання додаткових зовнішніх інструментів.

## Запуск застосунку через PM2

Для керування Node.js-застосунком було використано менеджер процесів PM2. Він дозволяє запускати сервер у фоновому режимі, переглядати список процесів, логи та моніторити використання ресурсів.

Для запуску застосунку використовується команда:

```bash
pm2 start server.js
```

Для перегляду списку процесів:

```bash
pm2 list
```

Для перегляду логів:

```bash
pm2 logs
```

Для відкриття панелі моніторингу:

```bash
pm2 monit
```

PM2 є корисним інструментом для production-середовища, оскільки дозволяє автоматично перезапускати застосунок у разі збою та контролювати його стан.

## Опис роботи застосунку

У межах лабораторної роботи було розширено серверний застосунок для системи оренди велосипедів. До вже існуючого REST API було додано технічні можливості, необхідні для стабільної роботи серверного застосунку.

Після запуску сервера всі HTTP-запити автоматично логуються за допомогою Morgan. Додатково події застосунку та помилки записуються у файли за допомогою Winston. Це дозволяє аналізувати дії користувачів, помилки сервера та продуктивність API.

Авторизований користувач може оновити свій аватар через endpoint `/auth/profile/avatar`. Для цього він надсилає PATCH-запит із файлом у полі `avatar` та JWT-токеном у заголовку `Authorization`. Сервер перевіряє токен, приймає файл, перевіряє його тип і розмір, зберігає у папці `uploads/avatars`, після чого оновлює поле `avatar` у таблиці `users`.

Також користувач може отримати інформацію про стан сервера через endpoint `/status`. У відповіді повертається uptime, використання пам’яті та CPU. Це дозволяє виконувати базовий моніторинг без встановлення додаткових сервісів.

У разі виникнення помилок вони обробляються централізовано через middleware. Помилка записується у файл логів, а клієнт отримує JSON-відповідь із повідомленням про помилку.

## Приклади тестування API

Для перевірки стану сервера використовується запит:

```bash
curl http://localhost:3000/status
```

Для тестування помилки:

```bash
curl http://localhost:3000/test-error
```

Для завантаження аватара користувача:

```bash
curl -X PATCH http://localhost:3000/auth/profile/avatar \
  -H "Authorization: Bearer YOUR_ACCESS_TOKEN" \
  -F "avatar=@./avatar.jpg"
```

Для перевірки профілю користувача:

```bash
curl http://localhost:3000/auth/profile \
  -H "Authorization: Bearer YOUR_ACCESS_TOKEN"
```

У результаті профіль користувача повертається разом із полем `avatar`.

У ході виконання лабораторної роботи було досягнуто поставленої мети. Було вивчено розширені можливості Node.js-додатків та реалізовано їх у серверному застосунку для системи оренди велосипедів.

У межах роботи було підключено логування HTTP-запитів за допомогою Morgan, реалізовано файлове логування подій і помилок через Winston, створено middleware для вимірювання часу відповіді та централізованої обробки помилок. Також було реалізовано endpoint `/status`, який дозволяє переглядати базові показники роботи сервера, зокрема uptime, використання пам’яті та CPU.

Окремо було реалізовано завантаження аватара користувача через Multer. Для цього в таблицю `users` було додано поле `avatar`, а в API створено захищений endpoint для оновлення аватара профілю. Сервер перевіряє тип і розмір файлу, зберігає його у папку `uploads/avatars` та оновлює дані користувача в базі даних.

Отже, лабораторна робота дозволила закріпити практичні навички розробки більш надійного Node.js-застосунку, який підтримує логування, обробку помилок, завантаження файлів і базовий моніторинг продуктивності.
