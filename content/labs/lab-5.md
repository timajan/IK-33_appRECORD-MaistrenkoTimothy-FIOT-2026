# Лабораторна робота №5

**З дисципліни:** «WEB-орієнтовані технології. Backend розробки»  
**Тема:** «Безпека та продуктивність серверних додатків. Безпека Node.js-додатків. Оптимізація запитів і кешування. Тестування API»

## Мета роботи

Метою лабораторної роботи є вивчення способів підвищення безпеки та продуктивності серверних застосунків на базі Node.js та Express. У межах роботи необхідно набути практичних навичок захисту REST API, валідації вхідних даних, обмеження кількості запитів, використання кешування, документування API та автоматизованого тестування.

Також метою роботи є вдосконалення вже створеного REST API для системи оренди велосипедів шляхом додавання механізмів безпеки, Redis-кешування, оптимізації GET-маршрутів, Swagger-документації, Docker-контейнеризації та навантажувального тестування.

## Короткі теоретичні відомості

Node.js є платформою для створення серверних застосунків на JavaScript. У поєднанні з Express він дозволяє будувати REST API, обробляти HTTP-запити, підключати middleware, працювати з базами даних і реалізовувати бізнес-логіку серверної частини застосунку.

Одним із важливих напрямів розробки backend-застосунків є безпека. Серверний застосунок повинен захищати дані користувачів, перевіряти вхідні дані, обмежувати підозрілу активність і не розкривати зайву службову інформацію. Для цього використовуються middleware, які додають захисні HTTP-заголовки, обмежують кількість запитів, перевіряють JWT-токени та виконують валідацію даних.

Для захисту HTTP-заголовків у Node.js часто використовується бібліотека **Helmet**. Вона додає набір безпечних заголовків, які зменшують ризик деяких типових атак, наприклад XSS або MIME sniffing.

Для обмеження кількості запитів використовується **express-rate-limit**. Цей middleware дозволяє встановити максимальну кількість запитів з одного клієнта за певний проміжок часу. Такий підхід допомагає зменшити ризик brute-force атак і надмірного навантаження на сервер.

Валідація даних є ще одним важливим елементом безпеки. Вона дозволяє перевіряти тіло запиту, параметри маршруту та query-параметри до того, як ці дані будуть використані в логіці застосунку або SQL-запитах. У роботі для цього використано бібліотеку **express-validator**.

Продуктивність API залежить від швидкості виконання запитів, кількості звернень до бази даних, розміру відповіді та використання кешування. Якщо endpoint часто повертає однакові дані, доцільно зберігати результат у кеші. Повторний запит у такому випадку може бути оброблений без звернення до бази даних.

Для кешування в роботі використано **Redis**. Redis — це швидке in-memory сховище типу key-value, яке часто застосовується для кешування відповідей API, сесій, тимчасових токенів і проміжних результатів обчислень. Якщо Redis недоступний, у реалізованому застосунку передбачено fallback на in-memory cache.

Для тестування API використовуються **Jest** та **Supertest**. Jest відповідає за запуск тестів, а Supertest дозволяє надсилати HTTP-запити до Express-застосунку без ручного запуску сервера в браузері або Postman.

Для документування API використано **Swagger**. Він дозволяє описати endpoints, параметри запитів, формати відповідей і перевіряти API через веб-інтерфейс.

## Постановка задачі

У межах лабораторної роботи необхідно розширити функціональність існуючого REST API для системи оренди велосипедів. Основна увага приділяється підвищенню безпеки, оптимізації продуктивності та тестуванню серверного застосунку.

Необхідно реалізувати:

- захист HTTP-заголовків за допомогою Helmet;
- обмеження кількості запитів за допомогою rate limiting;
- валідацію вхідних даних;
- JWT-автентифікацію для захищених маршрутів;
- перевірку ролей для адміністративних операцій;
- кешування GET-запитів через Redis;
- fallback на in-memory cache, якщо Redis недоступний;
- оптимізацію маршруту отримання списку велосипедів;
- Swagger-документацію API;
- Docker-контейнеризацію застосунку;
- автоматизоване тестування через Jest і Supertest;
- навантажувальне тестування через Artillery;
- аналіз продуктивності до та після оптимізації.

Практична реалізація виконується в межах існуючого застосунку `rentBikeApi`, який уже містить моделі користувачів, велосипедів, станцій, бронювань, маршрути автентифікації та підключення до бази даних MySQL через Sequelize.

## Реалізація захисту API

Для базового захисту серверного застосунку у файлі `app.js` було підключено кілька middleware. Вони відповідають за безпечні HTTP-заголовки, CORS, стиснення відповідей, обмеження розміру тіла запиту та rate limiting.

Приклад налаштування middleware у `app.js`:

```js
app.use(helmet());
app.use(cors({ origin: process.env.CORS_ORIGIN || '*' }));
app.use(compression());
app.use(express.json({ limit: '1mb' }));
app.use(express.urlencoded({ extended: true, limit: '1mb' }));
app.use(limiter);
```

Helmet автоматично додає захисні HTTP-заголовки. Наприклад, під час тестування було перевірено наявність заголовка:

```text
X-Content-Type-Options: nosniff
```

Це означає, що сервер вказує браузеру не виконувати MIME sniffing і не намагатися інтерпретувати файл як інший тип контенту.

Також було додано обмеження розміру JSON-запиту до `1mb`. Це дозволяє зменшити ризик надсилання занадто великих запитів і перевантаження сервера.

## Реалізація rate limiting

Для захисту від великої кількості запитів було використано бібліотеку `express-rate-limit`. У застосунку створено middleware `limiter`, який обмежує кількість запитів за певний проміжок часу.

Приклад реалізації:

```js
const limiter = rateLimit({
  windowMs: Number(process.env.RATE_LIMIT_WINDOW_MS || 60_000),
  max: Number(process.env.RATE_LIMIT_MAX || 100),
  standardHeaders: true,
  legacyHeaders: false,
  message: {
    error: 'Too many requests, please try again later',
  },
});
```

За замовчуванням клієнт може виконати до 100 запитів за 60 секунд. Якщо цей ліміт перевищено, сервер повертає помилку з повідомленням:

```json
{
  "error": "Too many requests, please try again later"
}
```

Параметри `RATE_LIMIT_WINDOW_MS` і `RATE_LIMIT_MAX` винесено у змінні середовища, тому їх можна змінювати без редагування коду.

## Реалізація валідації даних

Для перевірки вхідних даних було використано бібліотеку `express-validator`. Валідація була додана до маршрутів велосипедів, станцій, бронювань і реєстрації користувача.

Для централізованої обробки помилок валідації створено файл `middleware/validateRequest.js`.

Приклад middleware:

```js
const { validationResult } = require('express-validator');

module.exports = (req, res, next) => {
  const errors = validationResult(req);

  if (!errors.isEmpty()) {
    return res.status(400).json({
      errors: errors.array(),
    });
  }

  next();
};
```

Після цього middleware можна підключати до будь-якого маршруту після списку правил валідації.

Наприклад, для створення велосипеда перевіряються такі поля:

```js
const bikeValidation = [
  body('title').trim().isLength({ min: 2, max: 100 }),
  body('type').trim().isLength({ min: 2, max: 50 }),
  body('price_per_hour').isFloat({ min: 0 }),
  body('status').optional().isIn(['available', 'rented', 'maintenance', 'inactive']),
  body('station_id').optional().isInt({ min: 1 }),
];
```

Якщо користувач надсилає некоректні дані, сервер не виконує основну логіку маршруту, а одразу повертає помилку `400 Bad Request`.

## JWT-автентифікація та перевірка ролей

У застосунку вже була реалізована JWT-автентифікація. У межах лабораторної роботи вона використовується для захисту маршрутів, які змінюють дані. Користувач повинен передати токен у заголовку `Authorization`.

Приклад заголовка:

```text
Authorization: Bearer YOUR_ACCESS_TOKEN
```

Для адміністративних операцій було додано перевірку ролі `admin`. Наприклад, створювати, редагувати та видаляти велосипеди може лише авторизований користувач з роллю адміністратора.

Приклад захищеного маршруту:

```js
router.post(
  '/',
  authMiddleware,
  roleMiddleware('admin'),
  bikeValidation,
  validateRequest,
  async (req, res) => {
    const bike = await Bike.create(req.body);
    await delByPrefix('cache:GET:/bikes');
    res.status(201).json(bike);
  }
);
```

Такий підхід дозволяє розділити доступ для звичайних користувачів і адміністраторів.

## Реалізація кешування через Redis

Для кешування GET-запитів було створено окремий модуль `utils/cache.js`. Основним варіантом кешування є Redis. Якщо Redis недоступний або вказано `CACHE_DRIVER=memory`, застосунок автоматично використовує in-memory cache.

Підключення Redis виконується через змінну середовища:

```env
REDIS_URL=redis://redis:6379
CACHE_TTL_SECONDS=60
```

Основні функції модуля кешування:

- `connectRedis()` — підключає Redis;
- `cacheMiddleware()` — middleware для кешування GET-відповідей;
- `getCache()` — отримує значення з кешу;
- `setCache()` — записує значення в кеш;
- `delByPrefix()` — очищає кеш за префіксом.

Приклад middleware кешування:

```js
function cacheMiddleware(ttlSeconds = DEFAULT_TTL) {
  return async (req, res, next) => {
    if (req.method !== 'GET') return next();

    const key = buildKey(req);
    const cached = await getCache(key);

    if (cached) {
      res.set('X-Cache', 'HIT');
      return res.json(cached);
    }

    const originalJson = res.json.bind(res);
    res.json = (body) => {
      setCache(key, body, ttlSeconds);
      res.set('X-Cache', 'MISS');
      return originalJson(body);
    };

    next();
  };
}
```

Для перевірки кешу використовується заголовок відповіді:

```text
X-Cache: MISS
```

Це означає, що дані були отримані з бази даних і збережені в кеш.

Повторний запит повертає:

```text
X-Cache: HIT
```

Це означає, що відповідь була отримана з кешу без повторного звернення до бази даних.

## Очищення кешу після зміни даних

Щоб користувач не отримував застарілі дані, після створення, редагування або видалення записів кеш очищається. Для цього використовується функція `delByPrefix()`.

Наприклад, після створення велосипеда очищається кеш усіх GET-запитів, які починаються з `/bikes`:

```js
const bike = await Bike.create(req.body);
await delByPrefix('cache:GET:/bikes');
res.status(201).json(bike);
```

Такий підхід дозволяє поєднати переваги кешування з актуальністю даних. Повторні GET-запити працюють швидше, але після зміни даних старий кеш видаляється.

## Оптимізація маршруту `GET /bikes`

Одним із головних завдань лабораторної роботи була оптимізація одного з маршрутів API. Для цього було обрано маршрут:

```text
GET /bikes
```

До оптимізації він міг повертати всі записи без обмежень і кожного разу звертався до бази даних. Після оптимізації було додано:

- пагінацію;
- фільтрацію за статусом;
- фільтрацію за типом велосипеда;
- пошук за назвою;
- вибір тільки потрібних полів;
- обмежене підключення інформації про станцію;
- кешування відповіді.

Приклад реалізації:

```js
const page = Number(req.query.page || 1);
const limit = Number(req.query.limit || 10);
const offset = (page - 1) * limit;

const where = {};
if (req.query.status) where.status = req.query.status;
if (req.query.type) where.type = req.query.type;
if (req.query.search) {
  where.title = { [Op.like]: `%${req.query.search}%` };
}

const { rows, count } = await Bike.findAndCountAll({
  where,
  include: [{ model: Station, attributes: ['id', 'name', 'address'] }],
  attributes: ['id', 'title', 'type', 'price_per_hour', 'status', 'description', 'photo', 'created_at'],
  order: [['id', 'DESC']],
  limit,
  offset,
});
```

У відповіді сервер повертає не лише список велосипедів, а й інформацію про пагінацію:

```json
{
  "source": "database",
  "pagination": {
    "page": 1,
    "limit": 10,
    "total": 25,
    "pages": 3
  },
  "data": []
}
```

Завдяки цьому клієнт може завантажувати дані частинами, а сервер не повинен повертати всю таблицю одразу.

## Автоматизоване тестування API

Для тестування API було використано Jest і Supertest. Тести знаходяться у файлі:

```text
tests/app.test.js
```

У тестах перевіряється:

- доступність головного endpoint `/`;
- наявність security headers від Helmet;
- робота endpoint `/status`;
- валідація некоректних даних під час реєстрації;
- заборона доступу до профілю без JWT-токена.

Приклад тесту:

```js
test('GET / returns API metadata and security headers', async () => {
  const response = await request(app).get('/');

  expect(response.statusCode).toBe(200);
  expect(response.body.message).toBe('Bike rental API is running');
  expect(response.headers['x-content-type-options']).toBe('nosniff');
});
```

## Аналіз продуктивності до та після оптимізації

До оптимізації маршрут `GET /bikes` кожного разу звертався до бази даних і міг повертати велику кількість записів. Це збільшувало час відповіді та створювало додаткове навантаження на MySQL.

Після оптимізації було реалізовано кілька покращень:

- додано пагінацію, тому сервер повертає лише частину записів;
- додано фільтрацію, тому база даних обробляє точніші запити;
- додано пошук за назвою велосипеда;
- обмежено список полів, які повертаються у відповіді;
- обмежено дані пов’язаної станції;
- додано Redis-кешування повторних GET-запитів;
- додано заголовки `X-Cache: MISS` і `X-Cache: HIT` для перевірки джерела відповіді.

Порівняння поведінки API:

| Показник | До оптимізації | Після оптимізації |
|---|---|---|
| Звернення до БД | При кожному GET-запиті | Тільки при першому запиті або після очищення кешу |
| Обсяг відповіді | Може бути великим | Обмежений через `limit` і `attributes` |
| Пагінація | Відсутня | Реалізована через `page` і `limit` |
| Фільтрація | Обмежена або відсутня | Реалізована за `status`, `type`, `search` |
| Кешування | Відсутнє | Redis або in-memory cache |
| Перевірка кешу | Відсутня | Через заголовок `X-Cache` |

Приклад першого запиту:

```bash
curl -i "http://localhost:3000/bikes?page=1&limit=10&status=available"
```

У відповіді буде заголовок:

```text
X-Cache: MISS
```

Приклад повторного запиту:

```bash
curl -i "http://localhost:3000/bikes?page=1&limit=10&status=available"
```

У відповіді буде заголовок:

```text
X-Cache: HIT
```

Це показує, що повторний запит обробляється швидше, оскільки дані беруться з кешу, а не з бази даних.

## Опис роботи застосунку

У межах лабораторної роботи було вдосконалено серверний застосунок для системи оренди велосипедів. Після запуску сервер підключає middleware безпеки, налаштовує обмеження кількості запитів, підключає Swagger-документацію, маршрути автентифікації, велосипедів, станцій, бронювань і endpoint стану сервера.

Користувач може реєструватися та авторизуватися через маршрути `/auth/register` і `/auth/login`. Після авторизації він отримує JWT-токен, який використовується для доступу до захищених маршрутів. Якщо токен відсутній або некоректний, сервер повертає помилку авторизації.

Адміністратор може створювати, редагувати та видаляти велосипеди і станції. Для цього запит повинен містити JWT-токен, а користувач повинен мати роль `admin`. Перед виконанням операції сервер перевіряє вхідні дані через `express-validator`.

GET-запити до списку велосипедів, станцій і бронювань кешуються. Перший запит отримує дані з бази даних і записує їх у Redis. Повторний запит з такими самими параметрами повертається з кешу. Якщо дані були змінені, відповідний кеш очищається.

Документацію API можна переглянути через `/api-docs`. Для перевірки стану сервера використовується `/status`, який повертає uptime, використання пам’яті та CPU.

- перевірку ролі `admin` для адміністративних операцій;
- Redis-кешування GET-запитів;
- fallback на in-memory cache;
- очищення кешу після зміни даних;
- оптимізацію маршруту `GET /bikes`;

Отримані результати показують, що серверний застосунок став більш захищеним, продуктивним і зручним для тестування та подальшого розвитку.

```bash
curl -X POST http://localhost:3000/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"test@example.com","password":"qwerty123"}'
```

```bash
curl http://localhost:3000/auth/profile \
  -H "Authorization: Bearer YOUR_ACCESS_TOKEN"
```

```bash
curl -i "http://localhost:3000/bikes?page=1&limit=10&status=available"
```

```bash
curl -i "http://localhost:3000/bikes?search=city"
```

```bash
curl -X POST http://localhost:3000/bikes \
```

## Висновок

У ході виконання лабораторної роботи було досягнуто поставленої мети. Було вивчено та реалізовано основні підходи до підвищення безпеки та продуктивності Node.js-застосунків.

У межах роботи було підключено Helmet для захисту HTTP-заголовків, реалізовано rate limiting для обмеження кількості запитів, додано валідацію даних через express-validator та використано JWT-автентифікацію для захисту приватних маршрутів. Також було додано перевірку ролі адміністратора для операцій створення, редагування та видалення даних.

Окремо було реалізовано кешування GET-запитів за допомогою Redis. Для підвищення надійності застосунку передбачено fallback на in-memory cache, якщо Redis недоступний. Після зміни даних кеш автоматично очищається, тому користувач отримує актуальну інформацію.

Маршрут `GET /bikes` було оптимізовано за рахунок пагінації, фільтрації, пошуку, вибору тільки необхідних полів і кешування. Це дозволило зменшити кількість звернень до бази даних і скоротити обсяг відповіді сервера.

Також було додано Swagger-документацію, Docker-контейнеризацію, сценарій навантажувального тестування Artillery та автоматизовані тести через Jest і Supertest. У результаті застосунок став більш безпечним, продуктивним, краще задокументованим і зручним для подальшого розвитку.
